### Finding #1 — CLI auth.login writes token to wrong TOML scope, breaking all CLI auth

**Section:** 1.1 / pre-flight (ground rule #2 — three surfaces)
**Severity:** Critical
**Surfaces:** CLI
**Status:** New

**Repro:**
1. Fresh `~/.forge/config.toml` (created by `forge init`)
2. `forge auth signup --email qa@forge.test --password 'QaTest!2026'`
3. `forge agents list`

**Observed:**
- `~/.forge/config.toml` ends with `[defaults]` and `forge auth login`/`signup` appends `api_key = "..."` after that. TOML scopes the new key under `[defaults]`, not at top-level.
- The CLI config loader (`cli/forge/config.py`) reads `[api].key` or top-level `api_key`, both empty.
- Every authenticated CLI command runs with no `Authorization` header; backend returns 422 for `Header(...)` requirements or 401 for `get_current_user`.

**Expected:**
The login token is the first thing the CLI loads, so it must land in a TOML position the loader actually reads. Both surfaces should reach the same authenticated state after `forge auth login`.

**Root cause:**
`auth_login` in `cli/forge/main.py` does a line-by-line search-and-append into the file, treating it as plain text. Any `[section]` ahead of the appended line scopes the key.

**Suggested fix:**
Write the token as `[api]\nkey = "..."` (the loader's primary lookup). Either rewrite the `[api]` block in place, or use `tomli_w` / a structured rewrite instead of `read_text().splitlines()` heuristics.

**Evidence:**
- Captured `~/.forge/config.toml` excerpt at finding time: `api_key` line was nested under `[defaults]` rather than `[api].key`. The CLI config loader returns empty for `get_api_key()`.
- Workaround applied for the rest of this run: a python snippet that lifts `api_key` out of `[defaults]` into `[api].key` after every login.

### Finding #3 — §1.1/1.2/1.4: web and CLI sessions are independent (architectural)

**Section:** 1.1, 1.2, 1.4
**Severity:** Pass (documented architectural behavior, not a bug per playbook ground rule)
**Surfaces:** Web, CLI

**Repro:**
- §1.1 CLI→web: `forge auth login --email qa@forge.test --password 'QaTest!2026'`, then navigate Playwright to `/dashboard` without filling the login form.
- §1.2 web→CLI: log in via web form, then `forge auth whoami`.
- §1.4 logout: log in on both, run `forge auth logout`, refresh `/dashboard`.

**Observed:**
- CLI persists its JWT to `~/.forge/config.toml` `[api].key`.
- Web persists JWT to `localStorage["forge_token"]` and cookie `sb-access-token`.
- These stores are entirely separate — neither surface reads the other's storage.
- §1.4: web stayed authenticated after `forge auth logout`; cookie + localStorage untouched.

**Decision (per playbook):**
The playbook explicitly says "If they're independent (separate Supabase session vs CLI JWT), document this honestly — it's a real architectural gap, not a bug to silently fix." Recording as documented behavior.

**Action:**
Add a new `docs/auth-model.md` capturing: same backend `/api/auth/login` endpoint, same JWT format, but two independent client-side stores. Cross-surface session sharing is not implemented and would require either:
- Reading `~/.forge/config.toml` from the web (security risk, browser can't anyway), or
- Issuing a short-lived "session-handoff" code (`forge auth web-link`) that the web can consume.

Logging this as a doc gap rather than a defect. The README implies parity but is silent on session sharing.

### Finding #4 — §1.3: JWT exp parity

**Section:** 1.3
**Severity:** Pass
**Surfaces:** Web, CLI, API

**Repro:**
1. Decoded the JWT from `~/.forge/config.toml` and from `localStorage.forge_token` after login.
2. Both surfaces use the same JWT-signed token from `/api/auth/login` (single source).

**Observed:**

**Note:**

### Finding #5 — Local-mode JWTs have no `exp` claim → tokens never expire

**Section:** 1.3
**Severity:** High (security)
**Surfaces:** API, CLI, Web

**Repro:**
1. `forge auth login --email qa@forge.test --password 'QaTest!2026'`
2. Decode the token: `python3 -c "import json,base64; t='<jwt>'; print(json.loads(base64.urlsafe_b64decode(t.split('.')[1] + '==')))"`

**Observed:**
- Decoded payload: `{"sub": "<user-id>", "email": "qa@forge.test"}` — no `exp`, no `iat`, no `nbf`.

**Expected:**
JWTs should expire (industry standard ~15min–24h with refresh-token rotation, or at least 30 days with explicit revocation). A token with no `exp` is valid forever unless the signing key changes.

**Suggested fix:**
In `backend/app/routers/auth_api.py`, add an exp claim to the payload (now + 7 days) in both signup and login. `get_current_user` already verifies expiry if the claim is present.

**Evidence:**
The captured decoded payload had only two keys (sub and email) and was missing both expiry-related claims.

### Finding #6 — `/api/dashboard/metrics` returns 500 on SQLite backend (ambiguous column)

**Section:** 2.1
**Severity:** Critical (the entire Live status section of the unified Dashboard is broken on SQLite)
**Surfaces:** API, Web

**Repro:**
1. SQLite backend (`FORGE_DB_BACKEND=sqlite`)
2. `curl -H "Authorization: Bearer <jwt>" http://localhost:8000/api/dashboard/metrics`

**Observed:**
- HTTP 500 `Internal Server Error`
- Backend exception: `sqlite3.OperationalError: ambiguous column name: created_at`
- Stack: `heartbeat_service.get_metrics` → SQLite `_exec_select` → join of `agent_heartbeats` and `agents` (via `agents!inner(user_id)`), both tables have `created_at`. The `gte("created_at", ...)` predicate is emitted as a bare column reference instead of `agent_heartbeats.created_at`.

**Expected:**
The SQLite backend's join handling (`backend/app/db/sqlite_backend.py`) should qualify column names with the primary table when an inner join is present, just as it does for the joined table.

**Suggested fix:**
In the WHERE-clause builder, prefix the primary table to bare column refs when `joins` is non-empty: `f"{primary_table}.{column}"`. Or thread the table-name into `_parse_single_filter`.

**Evidence:**
- `qa-evidence/finding-06-trace.txt` — full traceback

**Impact downstream:**
- `LiveStatusSection` (the new unified `/dashboard#live`) fetches `/api/dashboard/metrics` and displays raw error / falls back to zero values.
- `/api/dashboard/active` may have the same issue (similar join structure) — to be verified.

### Finding #7 — `POST /api/agents` returns 500 but row is already inserted (phantom writes)

**Section:** 6.1 (also blocks 2.1)
**Severity:** Critical
**Surfaces:** API, CLI

**Repro:**
1. Clean SQLite, no agents
2. `forge agents create --name "QA-Researcher" --prompt "..."` → 500
3. `curl /api/stats` → `total_agents` increments by 1 anyway

**Observed:**
- `ResponseValidationError: 1 validation errors: {'type': 'missing', 'loc': ('response', 'is_template'), 'msg': 'Field required'}`.
- The SQLite `insert(...)` returns `{name, description, system_prompt, tools, workflow_steps, model, user_id, id, created_at, updated_at}` — **no `is_template`** field even though the SCHEMA defines it with default False.
- `AgentResponse` requires `is_template`. Pydantic 500s on serialization.
- The DB row is already committed at this point — the user gets an error and the row leaks.

**Expected:**
Either the SQLite insert path applies schema defaults so all required response fields are populated, or the `AgentCreate` model includes `is_template` (default False) so it's part of the input. The serialization step should never see a row missing required columns.

**Suggested fix:**
Two-line backend fix in `backend/app/db/sqlite_backend.py` — when inserting, walk `SCHEMA[table]['columns']` and apply default values for any column not present in the input dict before writing. Verify the same behavior on Supabase (Postgres handles defaults at the DB level, so SQLite needs to emulate).

**Blast radius:**
This breaks **every** agent create from CLI or API. Web (Playwright) creates work because they POST through a different code path (likely sets is_template=False explicitly). Confirms this is SQLite-specific schema-default leakage.

**Evidence:**
- `qa-evidence/finding-07-trace.txt` — full traceback from `/tmp/forge-uvicorn.log`

### Finding #9 — `forge config set-default-model` writes wrong key, `show` always returns empty

**Section:** 4.1
**Severity:** Medium
**Surfaces:** CLI
**Status:** Fixed in PR #61 — writer now updates `[defaults].model` in place; loader tolerates legacy `default_model`.

**Repro:**
1. `forge config set-default-model "ollama/llama3.2:3b"` → "Default model set to: ollama/llama3.2:3b"
2. `cat ~/.forge/config.toml` → `[defaults]` block has `default_model = "ollama/llama3.2:3b"`
3. `forge config show` → `default_model:` (empty)

**Root cause:**
- `config_set_default_model` (cli/forge/main.py) writes `default_model = "..."` to the `[defaults]` section.
- `get_config()` (cli/forge/config.py) reads `[defaults].model` (line 41) — looks for `model`, not `default_model`. Mismatched key.

**Suggested fix:**
Either make the loader fall back to `default_model` (matches the older flat-file `default_model` top-level key it already accepts), or have the writer write to `[defaults].model` instead. Pick one canonical key and make set + show + load all use it.

### Finding #26 — `DELETE /api/agents/<id>` 500s on FOREIGN KEY constraint

**Section:** 6.4
**Severity:** Critical
**Status:** Fixed in PR #66 — route now nulls back-pointers on history tables and drops transient heartbeats; runs survive with agent_id=null, schema also updated for fresh installs.
**Surfaces:** API, CLI

**Repro:**
1. Create an agent (`forge agents create`).
2. Run it once (any input) so a `runs` row references it.
3. `DELETE /api/agents/<id>`.

**Observed:**
- 500. Backend log: `sqlite3.IntegrityError: FOREIGN KEY constraint failed`.
- The `runs` table FK to `agents.id` is configured as RESTRICT, not SET NULL or CASCADE.

**Expected (per playbook §6.4):**
"Existing runs of that agent: still visible in Runs history (don't cascade-delete)" — so the FK should be `ON DELETE SET NULL` (or the route should soft-delete the agent rather than physically remove the row).

**Suggested fix:**
Migrate the `runs.agent_id` FK to `ON DELETE SET NULL`. Same for any other table that references `agents.id` (heartbeats, token_usage). Or switch the agents table to use a `deleted_at` column instead of physical deletes.

### Finding #27 — `forge costs --period today` reports 0 even when runs have token usage

**Section:** 3.1
**Severity:** High
**Status:** Fixed in PR #67/#68 — agent executor now writes to token_usage so /api/costs/* aggregates see the data.
**Surfaces:** CLI, API

**Repro:**
1. Run an agent that produces tokens (e.g., `forge agents run <id> --input 'say hello'`).
2. `/api/stats` correctly reports `total_tokens > 0`.
3. `forge costs --period today` reports 0 input/output/requests.

**Root cause:**
`/api/stats` queries the `runs` table; `forge costs` queries `/api/costs/summary` which queries the `token_usage` table. They're separate. The agent executor populates `runs.tokens_used` but the path that writes to `token_usage` is gated on a code branch that's apparently not hit for Ollama runs.

**Suggested fix:**
Either backfill `token_usage` rows from the agent executor for every provider (so the cost endpoints can aggregate), or reduce the cost endpoints to query `runs.tokens_used` directly when no `token_usage` rows exist.

### Finding #31 — Default model env-var fallback isn't applied at run-time

**Section:** 4.3
**Severity:** Medium
**Status:** Fixed in PR #66 — registry resolves unrouteable seeded defaults to the default provider's own default_model.
**Surfaces:** API, CLI

When an agent is created without an explicit `model` field, the run-time
resolver uses `ProviderRegistry._default_model` — which is hard-coded to
`gpt-4o-mini` (or `$DEFAULT_MODEL`) at registry init. So even with
`forge config set-default-model "ollama/qwen2.5:7b-instruct"` (CLI default)
or `DEFAULT_MODEL=ollama/qwen2.5:7b-instruct` (env), agents created without
`model` still try `gpt-4o-mini` and 404 against Ollama.

**Suggested fix:**
On agent run, when `agent.model` is None, look up the user's preference from
the providers config (or fall back to `default_provider/<first_model>`)
rather than the hard-coded constant. The `/api/providers` endpoint in PR #61
already does this; the run path should use the same logic.
